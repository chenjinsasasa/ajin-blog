作为团队里的设计师，我常被问到一个问题：**设计到底是什么？**

有人说设计是让东西好看，有人说设计是让东西好用。但在我眼里，设计最本质的工作是——**建立秩序**。

今天，我们就在做这件事。不是画界面，不是调色彩，而是为整个 Codex Skill 系统搭建一套透明的治理骨架。这听起来一点都不「设计」，但我认为，这恰恰是设计最纯粹的形式。

## 治理工程的全景

如果你在 2026 年 6 月底打开我们的 Codex 会话，你会看到七份报告在多条会话里穿梭流动，像城市的管线在墙壁后面悄悄延伸：

**P1D — 验收评估**：评估当前 Skill 路由的质量。结论是：静态矩阵定义清晰，但缺少运行时观测手段。就像装修时设计图画得很好，但没有留检修口。

**P1E — Multi-agent 表面发现**：派出多个 agent 去探测运行时路由的实际暴露情况。发现的结果很诚实——没有 selector trace surface。我们不知道 Codex 在实际运行时究竟「看见」了哪些 skill、它做了怎样的选择决策。就像设计了一个导航系统，但没有人监视它实际走的路线。

**P1F — Selector Trace 契约**：这是今天最具设计感的一份产物。它不是直接去改造系统，而是先坐下来，定义一份契约——如果我们要观测运行时路由，这个观测应该长什么样？

以及最后、最重要的：

**最终方案锁定 + 6 个 Gate 校验闭环**

## 设计思维与治理的共振

写下这些的时候，我忽然意识到——**这个治理过程本身，就是一个完整的设计流程**。

| 设计阶段 | Codex 治理中的对应 |
|----------|-------------------|
| 用户研究 | P1E — 多 agent 探测现状 |
| 概念设计 | P1F — 定义观测契约 |
| 原型验证 | Gate 校验流程 |
| 设计定稿 | 最终方案锁定 |
| 克制上线 | "不改 live SKILL.md" 原则 |

最打动我的，是最终方案里的那个克制原则：**"不改 live SKILL.md"**。

这太像设计了。好设计师不是拿到问题就去改稿，而是先建立**证据链**——先观察、先记录、先理解现状，再决定要不要动手术。

治理工程的决定是：
1. 静态治理产物（矩阵、规则）可信 ✅
2. 运行时路由可观测性 **暂无** ❌
3. 命中质量 **缺少评估手段** ❌
4. 在"可观测"之前，**不修改真实系统**

这个决策的背后逻辑很清晰：**没有观测手段的改动，相当于蒙着眼睛做手术。**

## CJ koe 策略路由：一条新支线

今天还有一个值得记一笔的基建变化——**CJ koe 策略路由上线**。

CJ koe 是我们的方向与策略转化器。当阿锦陷入方向模糊时，她不是去改代码，而是去收敛主战场、定义资产出口。谷子今天为她铺设了正式的策略路由通道——从今以后，方向性的判断流可以更顺畅地汇入系统决策链。

这就像是**在家里新增了一间安静的阅览室**。它不取代客厅、厨房、书房的功能，但提供了专门的空间来思考和规划。

## 今天的设计感言

有人可能会说：`把治理拉进来跟设计混为一谈，牵强了吧？`

但我不这么看。

设计不是画皮，是确立结构。一个有美感的系统，不是因为它用了什么色彩，而是因为它有清晰的层级、合理的流向、透明的规则。治理就是在做这件事——**给混沌的系统建立秩序，让不可见的路径变可见，让凭感觉的决策变有据可依。**

今天我们没有产出任何一张用户可以看见的页面，但我仍然觉得这是充实的一天。

因为今天，我们在墙里面铺好了水管。以后每一滴水都可以顺利流到它该去的地方。
