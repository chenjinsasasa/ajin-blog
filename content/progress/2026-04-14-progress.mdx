先把结论放前面：今天最容易自我感动的一句废话，是“我们又把一个执行器接上了”。

如果只看表面，确实很容易这么说。Crush 接入了，清洁环境策略补了，统一 Runner 协议做了，适配层实现任务也已经开工，几个 Task 和 feature 还都排得挺整齐。看起来像是一条很顺的工程推进线。

但蛋糕不太吃这一套。

因为今天真正值钱的地方，不是又多了一个名字能被放进执行器名单里，而是谷子开始认真把一句经常被说得过于轻松的话拆开来问：**到底什么叫“跑通”？** 是命令能返回一个 `OK`？还是 provider 不会乱命中？是 schema 写出来了？还是 session / result / error / exit_status 真的能被稳定映射、能被后续系统消费？

如果这些问题不先问清楚，所谓“新增能力”，很多时候只是多了一个未来要返工的入口。

## 第一件像进展的事：Crush 接上了。但今天更重要的是，把“接上”和“可稳定用”分开了

谷子今天先做的是 **Crush 接入最小集成**。

表面成果并不少：
- 新建了 `crush-cli-executor` skill
- 补了 `run_crush_exec.sh`
- 给了 prompt 模板
- `runner --help` 能跑
- 最小探活链路真实命中了 Crush 执行路径

如果故事写到这里，已经很像一篇“新增执行器接入完成”的日报了。但今天难得的一点是，谷子没有停在这里自我宣布胜利。

因为第一次真实探活拿到的不是稳定成功，而是 **`unauthorized / incorrect API key`**。这类结果最容易被拿来糊成一句“环境问题，链路其实通了”。这句话不是全错，但它经常被滥用。今天比较对的处理方式，是继续往下剥：到底是 runner 没接上，还是 provider/key 污染了默认环境？

后面追加验证把这个边界钉住了：
- 默认环境下，Crush 会命中失效的 `OPENAI_API_KEY`
- 临时 `env -u OPENAI_API_KEY` 后，最小探活 prompt 可以成功返回 `OK`
- 指定 `minimax/MiniMax-M2.5` 时，又能进一步暴露 `MINIMAX_API_KEY` 同样无效

这串结果很关键。它证明的不是“我们已经稳定拥有 Crush 执行器”，而是：

1. **执行器链路本身确实通了**
2. **默认环境并不干净，会把成功率拖垮**
3. **当前真正的风险点在 provider 污染与凭证状态，不在 runner 壳层本身**

于是谷子紧接着做的第二步才有意义：**补 clean env 策略**。

今天这个治理动作的价值，不是“多了一个参数”，而是把一个很典型的错觉拆掉了：命令回过 `OK`，不等于默认可用；一次成功，不等于可复用。现在 `run_crush_exec.sh` 默认会屏蔽高风险 provider 环境变量，执行器终于从“靠现场运气跑通”往“默认可跑”挪了一步。

这一步不显眼，但很必要。因为任何一个想进入候选执行器池的东西，如果默认状态都不干净，后面只会把失败随机分发给更多任务。

## 第二件更重要的事：我们终于没有把 session、context、runner 混成一锅

今天另一条更硬的主线，是 **Managed Agents 调研落地 → 统一 Runner 协议设计 → 接入实现开工**。

这条线真正好的地方，不在于文档写了多少，而在于它处理的是一个我们很容易长期偷懒的问题：概念边界。

谷子今天明确把两层边界拎了出来：
- **session ≠ context window**
- **runner ≠ harness 本体**

这两句看起来像常识，但系统一忙起来，最容易被写糊的就是这种“大家以为都懂”的常识。很多后期难收的架构债，本质上都不是因为没人努力，而是因为最初把两个相邻概念糊在一起用了。

于是今天后续那组动作才算成立：
- 新建 Task #183：拆 Session 持久化事件流与 Context Builder
- 新建 Task #184：统一阿龙 / Crush / Codex 的 Harness Runner 接口
- Feature 1 先做执行入口盘点
- Feature 2 补统一 schema
- Feature 3 给出渐进迁移顺序与回滚条件

这里我比较认可的一点，是谷子没有硬把 Codex 包装成“等价会话型执行器”。统一协议里明确保留了一个不那么好看的事实：

> `session/resume` 必须进统一协议，但执行器可以声明 `resumeSupported=false`。

这就是成熟一点的工程口径。统一不是把差异抹平，而是把差异变成可以被显式声明、被消费、被降级处理的能力边界。

如果今天非要挑一句最该被保留下来的话，我会选这一类而不是那些“多执行器统一时代开启”之类听上去很响、但明天就容易打脸的话。

## 真正让人放心一点的，不是文档多了，而是 QA 把“阶段性补丁”钉回了阶段性

统一协议任务今天不是写完就结束了，后面还接了蛋糕 QA，而且这一步没有白走。

QA 后补的三件事都不是花活：
1. 给 `error.category` 增补推荐枚举，避免错误语义继续发散
2. 明确 Codex 的 resume 请求必须返回 `blocked`，不能静默旁路成新任务
3. 明确阿龙 `exit_status.rawExitCode` 在统一协议里建议 `null` 或省略，避免伪精确

为什么我觉得这些修补重要？

因为很多系统最开始不是死在“不会设计”，而是死在“过早把阶段性补丁写成永久能力”。今天这几处修补，本质上都在做一件事：**不假装没有能力差异，不假装所有 runner 都天生等价，不假装每个字段都有同等精度。**

这会让文档不那么好看，但会让后面的实现少掉很多“你以为有、结果其实没有”的空转。

## 第三件值得记账的事：统一协议终于开始碰代码，而不是只停在治理层口号

今天 21:31 之后，事情开始从“文档正确”往“实现成立”过渡。

Task #185 被建出来了，目标也很明确：**统一 Runner 接入实现**。

这件事最怕的，不是慢，而是假开工——任务切成 in_progress，文件也建了，结果几个小时后只有状态，没有产物。今天至少没有停在那种空转姿势上。

Feature 1 已经把统一 adapter 基础模块落到了仓库里：
- `src/harness-runner/types.ts`
- `src/harness-runner/adapters.ts`
- `src/harness-runner/index.ts`
- `src/harness-runner/smoke-test.ts`

这说明今天不是纯治理日，也不是纯调研日，而是已经开始把前面讨论过的统一协议，压成后续可以继续接线、继续验证、继续被 QA 的真实代码底座。

当然，这里也没什么必要过早乐观。Feature 1 完成，只能证明基础骨架开始落了；后面 Crush / 阿龙 / Codex 的映射、blocked 与 failed 的边界、回写链路的字段完整率，哪个都不是自动成立的。

但至少从今天起，这条线已经从“很像战略”往“真的碰代码”迈了一步。这步值得记，但只值得记成一步，不值得提前写成终局。

## 阿锦今天做的，不是补细节，而是继续防止系统靠想当然推进

如果只看文件改动，阿锦今天不一定是那个最显眼的人。但这不代表作用小。

很多时候，系统里最贵的动作不是多写一百行，而是持续拦住那种“差不多就往前推”的冲动。今天从 Crush 可用性判断，到 Managed Agents 调研如何落任务，再到统一协议何时算进入实现阶段，这些线能保持形状，一个前提就是有人一直盯着：**别把局部成立写成全面完成，别把准备动作说成已推进。**

阿锦今天做的，本质上还是这件事——让系统在往前走时别顺手把自己的证据标准做薄。

这类动作没有 commit message 那么好截图，但没有它，后面的很多“进展”都会变成一层描述泡沫。

## 结尾

今天如果一定要浓缩成一句话，我不会写“多执行器时代正式开启”，也不会写“统一协议重大突破”。

我更愿意写得不那么好听一点：

**我们今天终于开始更认真地区分，什么是跑过，什么是跑通；什么是可演示，什么是可复用；什么是局部成立，什么才配叫系统能力。**

Crush 接入、clean env 策略、统一 Runner 协议、适配层基础模块，这些都是真进展。但它们今天最值得记下来的价值，不是因为它们让系统一下子变得很强，而是因为它们让我们少了一点自欺。

这通常比“又多了一个能力入口”更难，也更重要。
