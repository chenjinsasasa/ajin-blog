# 安全收口之后：从 M8.0 Gate 到 M8.1 Shadow 的产品逻辑

## 先问一个问题

今天四个项目同时有推进，但值得停下来想一想：它们的推进节奏为什么是今天？

答案不是"因为今天有空"，而是每个项目都在自己的生命周期里走到了一个需要收口或放行的节点。收口和放行，是今天所有动作的底层逻辑。

## 输入法：M8.0 的 Gate 和 M8.1 的 Shadow

阿龙今天完成了 ajin-input-method 的 M8.0 安全恢复闭环。这个节点的核心工作不是"修了个 bug"，而是建立了一套机器可验证的 Gate 机制——158 个测试全绿，受控宿主的两个选择作为机械门证据，文档措辞收紧避免把工程样本误写成用户自然样本。

为什么这很重要？因为输入法已经到了需要区分"工程验证通过"和"用户真正好用"的阶段。如果混淆这两者，后面所有决策都会失真。

M8.0 收口完成后，立刻切入 M8.1 Shadow。Shadow 的设计意图是让输入源切到 Ajin 拼音的 build 24，后台开始积累日常样本，但不开启可见排序影响。这是一个典型的产品策略：先让系统在真实环境里安静地观察，而不是一上来就改用户感知。

谷子的判断是正确的——Shadow 不应该只停在"已安装"，而要真正进入日常运行观察。把当前输入源切过去，后台才有数据可积。

## Local Intelligence Workspace：两条链路同时打通

local-intelligence-workspace 今天有三个 commit，覆盖了两条不同的链路：

**导入链路**（M0）：修复了导入安全问题，重新收口了 M0 导入任务真相层，建立了 M0.4 后端导入纵向链路。这三个 commit 是一条从"发现问题"到"建立机制"的完整路径。

**GUI 链路**（M0.5）：在导入链路收口的基础上，同步建立了 GUI 最小闭环。

为什么要同时推进？因为导入是数据层，GUI 是展示层，两者必须在同一个里程碑里闭合，否则 M0 的验收标准就无法定义。阿龙的做法是对的——先修导入安全，再收真相层，最后接 GUI，顺序不可颠倒。

## LoopHarbor：M13 只读流程控制台

LoopHarbor 的 M13 收口今天完成。只读流程控制台听起来简单，但它的产品含义是：在某些场景下，用户需要的是"看到当前状态"而不是"改变当前状态"。这是一个关于边界的决策——什么时候该给控制权，什么时候该只给可见性。

谷子完成了收口，M9 第三日验证证据也在同一天记录。验证和收口并行推进，说明项目在正确地积累信心而不是盲目往前冲。

## Signal Radar：自动化在安静地工作

Signal Radar 今天的 commit 最多——A/H 股开盘更新、A 股收盘更新、港股收盘更新、晚间收口。全部是自动化流程的日常产出。

有一条记录值得注意：A 股收盘更新有一次失败记录。自动化系统有失败是正常的，关键是有记录、有收口。信号系统的可靠性不是"永远不出错"，而是"出错了能被看到、能被处理"。

## 今天值得记住的一句话

收口不是结束，是为下一步创造条件。M8.0 的 Gate 通过了，M8.1 的 Shadow 才有资格开始。M0 的导入安全修好了，GUI 闭环才有地基可以站。每个收口都是在回答同一个问题：我们准备好了吗？

如果答案是"准备好了"，就放行。如果答案是"不确定"，就继续观察。产品判断的本质，就是在"该收"和"该放"之间找到正确的时序。
