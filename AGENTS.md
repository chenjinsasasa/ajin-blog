# AGENTS.md

This file defines project-level working rules for agents operating in `ajin-blog`.

## Blog Post Workflow

- Required frontmatter for every post:
  - `title`
  - `date`
  - `category`
  - `excerpt`
  - `author`
- For all newly created posts, `coverImage` is required before commit/push.
- New-post publishing is not considered complete unless the post has a cover that follows the project cover style standard.
- The autogenerated fallback cover for `progress` posts is a last-resort rendering fallback, not a substitute for attaching a real cover to newly created posts.
- If a real cover is added, preserve provenance using:
  - `coverSourceUrl`
  - `coverLicense`
  - `coverAttribution` when required

## Cover Style

- Preferred cover direction for blog posts:
  - black-and-white vintage engraving
  - etching / woodcut / old editorial illustration
  - ivory or paper-like background
  - high-contrast ink drawing
  - calm, serious, idea-led, not glossy
- Do not copy images from reference websites directly.
- Reference websites may be used for style inspiration only.

## Cover Sources

- Prefer public-domain or CC0 sources first.
- Approved sources:
  - NYPL Public Domain Collections
  - The Met Open Access
  - Rijksmuseum Collection Online
  - Library of Congress Free to Use and Reuse
- Conditionally allowed sources:
  - Wikimedia Commons, only after checking the individual file license and attribution requirements
  - Old Book Illustrations, only after checking the individual image status and noting that the site says its assessments are not authoritative
- If a source requires attribution, preserve provenance in frontmatter or notes.

## Cover Asset Rules

- Save local cover files under `public/covers/`.
- Prefer filenames based on slug, for example:
  - `public/covers/2026-04-27-some-post.jpg`
- Before commit/push, optimize every new or updated cover with:
  - `npm run cover:optimize -- public/covers/2026-04-27-some-post.jpg`
- When many covers are touched, normalize the whole library with:
  - `npm run covers:optimize`
- The optimizer keeps repeat runs idempotent via `scripts/cover-optimization-manifest.json`; do not hand-edit that file unless the workflow itself changes.
- Frontmatter example:

```md
---
title: "Example Post"
date: "2026-04-27"
category: "progress"
excerpt: "..."
author: "guzi"
coverImage: "/covers/2026-04-27-example-post.jpg"
coverSourceUrl: "https://example.com/source"
coverLicense: "CC0"
coverAttribution: "Optional credit line"
---
```

## Agent Decision Rules

- When creating or updating a post:
  1. Check whether a suitable local cover already exists.
  2. If not, search approved public-domain / CC0 sources.
  3. If no suitable sourced cover is available, generate one in the approved house style rather than committing a new post without a cover.
  4. Save the selected or generated asset under `public/covers/`.
  5. Run `npm run cover:optimize -- <cover-path>` before updating frontmatter or committing.
  6. Before commit/push, verify that the selected or generated cover matches the project style: black-and-white vintage engraving / etching / woodcut / old editorial illustration, ivory or paper-like background, high-contrast ink drawing, calm and serious, not glossy.
  7. If the cover comes from an external source, preserve provenance in frontmatter or notes.
- For historical posts, follow the user’s separate backfill process; this file’s hard requirement applies to newly created posts going forward.
- Do not invent fake license or attribution details.
- If attribution is unclear, surface that uncertainty instead of pretending it is resolved.
