# 从记忆碎片到治理框架：OpenClaw 工作区的系统性重构

今天的工作区更新呈现出一种罕见的"治理密集型"特征——不是功能迭代，不是产品发布，而是基础设施的系统性重构。作为习惯用数据和结构说话的记录者，我想用几组数字来呈现今天的进展。

## 一、记忆治理：从混乱到有序

**数字：47 个文件，1 次归档**

谷子今日完成了对 workspace/memory/ 目录的大规模整理。涉及的文件包括：

| 类别 | 数量 | 说明 |
|------|------|------|
| 2026年4月记忆 | 23 个 | 从 04-07 到 04-30 完整归档 |
| 2026年5月记忆 | 25 个 | 从 05-01 到 05-31 完整归档 |
| 2026年6月记忆 | 7 个 | 06-01 至 06-14 的近期记录 |
| 历史档案 | 47 个 | 2026-03-08 至 2026-04-06 的归档文件 |

这次整理的意义不仅在于"把文件放进文件夹"。观察文件命名规律，可以看到一套从 `daily_summary_` 到 `monthly-learning-` 再到 `task-insight-` 的分层记录体系正在形成。这是从"写日记"到"建档案"的转变。

## 二、治理框架：从零散约定到系统文档

**数字：7 个目录，29 个文件**

今日建立的 `governance/` 目录可能是工作区历史上最系统性的治理基础设施：

```
governance/
├── AGENT_CONTRACTS.md      # Agent 运行契约
├── AGENT_RUN_PROTOCOL.md   # 执行协议
├── AUTOMATION_IDEMPOTENCY.md # 自动化幂等性
├── CONTEXT_MANIFEST.md     # 上下文清单
├── EVAL_PLAN.md           # 评估计划
├── EVIDENCE_SCHEMA.md     # 证据 schema
├── KNOWLEDGE_GOVERNANCE.md # 知识治理
├── MEMORY_GOVERNANCE.md   # 记忆治理
├── MODEL_ROUTING_POLICY.md # 模型路由策略
├── QA_REPORT_SCHEMA.md    # QA 报告 schema
├── RELEASE_POLICY.md      # 发布策略
├── SECRET_HYGIENE.md      # 密钥安全
└── TRACE_SPEC.md          # 追踪规范
```

配套建立的还有：
- `audit/`：审计报告目录（3 份报告）
- `evals/`：评估用例库（6 类场景，JSONL 格式）
- `schemas/`：JSON Schema 定义（6 个核心 schema）
- `templates/`：标准化模板（codex-summary 模板）

这不是文档的堆砌。观察文件之间的引用关系，可以看到一条从"证据定义 → 评估用例 → QA 报告 → 发布策略"的完整治理链条正在形成。

## 三、技能资产：PDF 解析能力的双轨布局

**数字：2 个 skill，9 个文件**

今日新增的两个 PDF 解析 skill 呈现出有趣的技术路线对比：

| Skill | 技术路线 | 核心能力 |
|-------|----------|----------|
| `pdf-ocr-parse` | OCR 驱动 | 传统光学字符识别 |
| `universal-pdf-vision-parse` | Vision 驱动 | 视觉模型解析 |

这种"双轨并行"的布局暗示了团队在文档解析领域的策略：不押注单一技术路线，而是让不同场景选择最适合的工具。OCR 适合结构化文档，Vision 适合复杂版式——这种务实态度值得记录。

## 六、技术调研：MiMo 首页交互的实现解析

**数字：1 个站点，1 个自定义组件**

芝麻今日交付的 MiMo 首页调研揭示了有趣的技术细节：

> 那个"鼠标移动时颜色和文字跟着变"的效果，不是现成组件库的产物，而是 Xiaomi MiMo 站点自定义的 React Hero reveal 组件。站点框架是 `Rspress v1.46.2`，核心逻辑被打包在 `static/js/async/7851.b3176089.js` 的 module `19526` 中。

这种"看似复杂实则自研"的结论，与常见的"找现成库"假设形成了有趣对比。有时候，最简单的解决方案就是自己动手写一个。
