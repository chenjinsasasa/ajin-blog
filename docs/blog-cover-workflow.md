# Blog Cover Workflow

For all newly created posts in this repo:

1. A real cover image is required.
2. `coverImage` must point to that local asset before commit/push.
3. The autogenerated fallback cover is only a rendering fallback, not a substitute for a new post cover.

## Target Style

For `progress` posts, default to:

- black-and-white engraving
- vintage etching
- woodcut illustration
- old editorial plate
- ivory paper background
- serious, restrained, concept-driven visual tone

Avoid:

- glossy SaaS illustration
- colorful 3D art
- neon gradients
- generic stock business photos
- direct screenshots from inspiration sites

## Source Priority

### Best Options

- [NYPL Public Domain Collections](https://www.nypl.org/research/resources/public-domain-collections)
- [The Met Open Access](https://www.metmuseum.org/hubs/open-access)
- [Rijksmuseum Collection Online / Data Services](https://data.rijksmuseum.nl/about/)
- [Library of Congress Free to Use and Reuse](https://lcweb2.loc.gov/free-to-use/)

### Use With Extra Checking

- [Wikimedia Commons reuse guidance](https://commons.wikimedia.org/wiki/Commons%3AReusing_content_outside_Wikimedia/en)
- [Old Book Illustrations terms](https://www.oldbookillustrations.com/terms-of-use/)

## Practical Policy

- If a strong CC0 / public-domain cover is easy to find, use it.
- If a source has per-file attribution requirements, keep those details with the post.
- If provenance is unclear, do not treat it as safe.
- If no good sourced cover is found quickly, generate one in the approved house style instead of publishing without `coverImage`.

## Local File Convention

- Save files to `public/covers/`
- Use slug-based filenames:
  - `/covers/2026-04-27-my-post.jpg`
  - `/covers/2026-04-27-my-post.png`

## Compression Workflow

- Optimize every new or updated cover before commit:
  - `npm run cover:optimize -- public/covers/2026-04-27-my-post.jpg`
- Normalize the full library when doing maintenance work:
  - `npm run covers:optimize`
- The optimizer currently:
  - keeps covers within a maximum frame of `1600x1200`
  - re-encodes images in place to reduce file size
  - records content hashes in `scripts/cover-optimization-manifest.json` so reruns skip unchanged files

## Frontmatter Convention

Minimum:

```md
---
title: "..."
date: "2026-04-27"
category: "progress"
excerpt: "..."
author: "guzi"
coverImage: "/covers/2026-04-27-my-post.jpg"
---
```

Recommended when provenance matters:

```md
---
title: "..."
date: "2026-04-27"
category: "progress"
excerpt: "..."
author: "guzi"
coverImage: "/covers/2026-04-27-my-post.jpg"
coverSourceUrl: "https://example.com/original"
coverLicense: "CC0 / Public Domain / CC BY ..."
coverAttribution: "Credit line if required"
---
```

## AI Generation Fallback

If no usable public-domain image is found, generate in this direction:

```text
black and white engraving, vintage etching, woodcut illustration, high contrast ink drawing, ivory paper background, editorial cover, dramatic but restrained, no modern color accents, no typography in image
```

## What Agents Should Do

When an agent writes a new blog post:

1. Write the post frontmatter first.
2. Search approved sources for a real cover.
3. If no suitable sourced cover is available, generate one in the approved house style.
4. Save the asset to `public/covers/`.
5. Run `npm run cover:optimize -- public/covers/<filename>`.
6. Add `coverImage` to frontmatter.
7. If the asset came from an external source, add `coverSourceUrl`, `coverLicense`, and `coverAttribution` when required.
8. Before commit/push, verify the final image still matches the house style after optimization.
