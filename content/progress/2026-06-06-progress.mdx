# Signal Radar 启动与 Agent 工作台的治理转向

今天的推进可以用一个关键词概括：**先文档，后代码**。这不是退缩，而是在多条线并行时，用文档作为对齐工具，降低后续返工成本。

---

## Signal Radar：从 0 到可运行

阿龙今天启动了 **signal-radar** 项目，完成了脚手架搭建（`b8ec295 chore: scaffold signal radar`）。这个项目的目标是做一个"每天一篇可阅读日报"的公网独立站。

但在启动过程中暴露了一个典型问题：**性能债务的早期积累**。

### 性能诊断数据

| 指标 | 数值 | 评估 |
|------|------|------|
| HTML 首包 | 865KB | 偏高 |
| 本地镜像目录 | 468MB | 过重 |
| 资产数量 | 556 个 | 过多 |
| 页面引用 | 772 个 asset refs | 冗余 |
| 图片 | 526 个 webp，约 29.1MB | 需优化 |
| JS | 4 个，约 3.1MB | 可接受 |

问题的根源在于采用了"完整 copy 原站源码"的策略。阿龙已经将 publications 页面从手写改为 BCG 源码本地镜像模式，同时解决了 3000 端口冲突（迁移到 3001）。

**关键判断**：468MB 的本地镜像在开发阶段尚可接受，但上线前必须瘦身。这不是技术债，是**数据债**——早期决策的副作用。

## 飞书/Hermes 接入：收敛为可复用治理模式

今天还处理了一个边界问题：**外部 Agent 接入的治理**。

阿龙读取了 feishu-cli/artifacts 目录下的 3 个文档：
- `agent-passport-protocol.md`
- `feishu-agent-integration.md`
- `hermes-bridge-spec.md`

判断结果是：**这些是治理层补丁级别的帮助，不是当前 MVP 必须立即实现的运行态材料**。

基于此，阿龙把 `Feishu/Hermes/芝麻` 这些实现细节收敛成当前项目可复用的**"外部 Agent 接入治理"**文档，没有把实现细节写成项目默认事实。

**关键原则**：外部系统的接入细节应该隔离在治理层，不应该污染项目的核心领域模型。

## 今日阅读：Anthropic 的化学家实验

今天还读了一篇 Anthropic 的研究文章 [Making Claude a chemist](https://wwwanthropic.com/research/making-claude-a-chemist)。

核心发现：
- Claude 可直接从期刊图片或手绘草图读取化学结构
- NMR 谱图预测精度：氢谱平均误差 ±0.079 ppm（远低于 ±0.20 容差窗口）
- 碳谱与专业软件 MestReNova 基本持平（±1.37 vs ±1.48 ppm）

**对我们的启示**：QA 评估应从"步骤合规"逐步转向"结果可接受性"。化学家最终关心的是"谱图预测是否落在正确区间"，而不是"模型是否按特定步骤执行"。
