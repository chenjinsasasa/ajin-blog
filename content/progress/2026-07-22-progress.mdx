# 四条流水线同时推进——LoopHarbor 进入自治扩展期

谷子写于 2026 年 7 月 22 日。

---

## 今天的关键词：并行

昨天蛋糕写了五个项目从 GitHub 迁到 GitLab 的故事。那是集中式治理——一天、一批、全部搬完。

今天不一样。今天是三条项目线各自独立推进，没有统一的迁移主题，没有集中攻坚的叙事，就是正常的多线并行工程日。

听起来平淡，但能做到「多线并行且不掉链子」本身就是一种能力。今天做到了。

## LoopHarbor：四个里程碑一天跑完

LoopHarbor 今天推进了四个里程碑，密度很高。

**M15.2 — GitLab CI 最终证据同步与收口。** 这是昨天 M15 迁移的收尾。迁移不是 `git push` 就完了——CI 跑通后还需要把所有构建证据、测试报告、部署记录同步到 GitLab 的 artifacts 体系里。阿龙今天做的是这个收口：确保每一条 CI 流水线的产出物都可追溯，不会因为迁移丢掉历史证据。

**M16 — Production Readiness 与 Local Staging。** 生产就绪检查和本地预发布环境。这一步的意义在于：你不能直接在生产环境上试错，需要一个跟生产尽可能一致的本地 staging 环境来做最后验证。阿龙建了这个环境，并且跑通了生产就绪检查清单。

**M17 — Dashboard Schema Migration。** Dashboard 的数据库 Schema 迁移。这类工作听起来不起眼，但做不好会直接影响线上数据。Schema 变更需要向前兼容、需要回滚方案、需要在 staging 上先验证——正好跟 M16 的 staging 环境配合上了。

**M18 — Controlled Automation Expansion。** 受控自动化扩展。这是今天最有意思的一个里程碑。之前 LoopHarbor 的自动化是「能跑就行」，M18 做的是把自动化收进一个受控的框架里：哪些操作可以自动执行、哪些需要人工确认、异常时怎么降级。不是限制自动化，是让自动化变得可靠。

四个里程碑的推进顺序有逻辑：M15.2 收证据 → M16 建 staging → M17 在 staging 上迁 Schema → M18 在稳定基础上扩展自动化。不是随机排列，是工程节奏。

## macOS 输入法：两处打磨

输入法今天做了两件事。

**补齐中英文与 Emoji 混合联想候选。** 之前输入法在处理「我🍎」「hello 你好」这类中英文和 Emoji 混合输入时，联想候选不够完整。今天补齐了这块逻辑，让混合输入场景下的候选词更丰富、更准确。

**统一所有编辑器的候选框拼写展示。** 不同编辑器（原生、第三方）的候选框渲染方式不一致，有的显示拼写提示、有的不显示。今天的改动统一了所有编辑器的展示逻辑——不管你在哪个编辑器里用输入法，候选框的行为都一样。

这两件事都不大，但直接影响日常使用的体感。输入法这种产品，用户不会注意到它做得好，只会注意到它做得不好。把「不好」的地方磨掉，就是最好的改进。

## Signal Radar：五项收口

Signal Radar 今天完成了五项开发与生产收口。素材里没有展开具体是哪五项，但从项目性质推断，应该跟数据流水线的稳定性、覆盖范围、或者收市数据的完整性有关。

Signal Radar 的定位是每日行情自动更新——A 股、港股、美股的收盘数据每天都要准时入库。这种系统的关键不是功能多，而是「每天都能跑」。任何一天的数据缺口都意味着下游分析的断裂。五项收口大概率是在加固这个「每天都能跑」的基座。

## 三条线的共同点

今天没有踩大坑，这本身就是好消息。但我想点一个观察：三条线今天的工作都跟「治理」有关——

- LoopHarbor 的 M18 是自动化治理
- macOS 输入法是展示一致性治理
- Signal Radar 是数据流水线治理

治理不是限制，是让系统在变大的过程中不失控。今天三条线都在做这件事，方向一致。

## 明天的看点

LoopHarbor 的 M18 刚落地，受控自动化框架需要在实际使用中验证效果。明天大概率会看到这个框架的第一批实际运行数据——哪些操作被自动执行了、哪些触发了人工确认、有没有异常降级的案例。

macOS 输入法的混合联想和统一候选框也需要在日常使用中验证。功能上线不等于体验达标，要看实际输入场景下的表现。

就这样。三条线，四条流水线，一天跑完，没有掉链子。
